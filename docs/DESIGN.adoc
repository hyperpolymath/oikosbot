This document records architectural decisions for the Oikos project.
Each decision is an ADR (Architectural Decision Record) with a status,

ADR statuses: `proposed` | `accepted` | `deprecated` | `superseded` | `rejected`

== DR-001 — DSL on Ephapax rather than a new language

*Date:* 2026-04-11 +
*Status:* accepted

=== Context

Stock-flow consistent macroeconomic modelling requires a type system capable
of expressing:

* *Linear* monetary values (money spent cannot be spent again)
* *Phantom type* parameters (currency as a compile-time dimension)
* *Region* scoping (values live only within a fiscal period)
* *Typestate* machines (financial instruments have lifecycle constraints)
* A *global structural invariant* (the Godley column-sum rule)

The Ephapax project (`hyperpolymath/ephapax`) was designed specifically to
provide all four of these mechanisms for a systems-programming setting:
linear types, Tofte–Talpin regions, and typestate.

The question was: build Oikos as (a) a standalone language with its own type
checker implementing the above, or (b) a surface grammar that desugars to
Ephapax IR and delegates type checking to Ephapax.

=== Decision

*Option (b): Oikos is a grammar and desugaring pass, not a language.*

Oikos defines a surface syntax designed for economists, a parser that
produces `oikos-syntax::Model`, and a desugaring pass
(`oikos-desugar`) that lowers `Model` to Ephapax IR.
Type checking is delegated entirely to the Ephapax type checker
(`ephapax-typing`).

=== Consequences

*Positive:*

* The formal semantic guarantees come from Ephapax, which has a Coq-verified
  type soundness proof.  Oikos inherits that proof for free.
* Oikos does not need its own type checker, region inference, or linearity
  analysis — substantial work avoided.
* Improvements to the Ephapax backend (better errors, WASM codegen,
  incremental compilation) automatically benefit Oikos users.
* The codebase boundary is clear: syntax and desugaring in Oikos,
  semantics in Ephapax.

*Negative / risks:*

* Oikos is blocked on Ephapax's parser milestone.
  The scaffold and spec can proceed independently; the desugaring pass
  cannot be end-to-end tested until `ephapax-parser` ships.
  This is tracked as a known blocker in `oikos-desugar/src/lib.rs`.
* Ephapax error messages may use systems-programming vocabulary that is
  unfamiliar to economists.  Oikos will need an error-translation layer
  that rewrites Ephapax type errors into economic language.
  This is deferred to a later ADR.
* If Ephapax changes its IR in a breaking way, Oikos must update its
  desugaring pass.  This coupling is acceptable given shared authorship.

=== Alternatives considered

* *Standalone language with custom type checker:* rejected.
  Duplicating region inference and linearity checking correctly is months
  of work; the Coq proof would need to be reproduced.
* *Dependently typed host (Idris2, Lean 4):* considered.
  Idris2 dependent types could encode the Godley invariant at the type level
  more directly.  Rejected for this iteration because the economist audience
  would face a steep learning curve with dependent-type error messages.
  Revisit post-v1.0 if Ephapax integration proves awkward.
* *Embedding in Julia (ModelingToolkit):* rejected.
  Julia/MTK does not enforce linearity or period scoping statically;
  violations would be runtime errors.  The core design requirement is
  compile-time rejection of invalid models.

== DR-002 — EUPL-1.2 licence

*Date:* 2026-04-11 +
*Status:* superseded by <<DR-010>>

=== Context

The hyperpolymath estate uses MPL-2.0 as its standard licence.
Oikos is intended for use by academic economists and public-sector
institutions, particularly within the European Union.

=== Decision

Oikos is licensed under EUPL-1.2 (European Union Public Licence).

=== Consequences

* EUPL-1.2 is an OSI-approved copyleft licence compatible with GPL-2.0,
  LGPL-2.1, AGPL-3.0, and several other copyleft licences.
* EUPL-1.2 has official EU translations in 23 languages, which is
  appropriate for a tool targeting European public-sector economists.
* MPL-2.0 is the standing estate default; this is an explicit
  exception.  The exception is recorded here and in the SPDX headers.
* Ephapax (MPL-2.0) is a dependency.
  PMPL compatibility with EUPL is an open question pending legal review.
  _Flag: confirm PMPL ↔ EUPL compatibility before v1.0 release._

[NOTE]
====
This decision was superseded by <<DR-010>> (2026-05-22) on the way to
v1.0, which adopts MPL-2.0 to remove the PMPL ↔ EUPL compatibility
unknown and align Oikos with the estate default.  The rationale below
is kept as a historical record.

== DR-003 — Godley invariant checking before desugaring

*Date:* 2026-04-11 +
*Status:* accepted

=== Context

The Godley column-sum rule is the most important accounting invariant.
It could be enforced in two places:

1. In `oikos-check` as a structural pre-check on the `GodleyMatrix` AST node.
2. In `ephapax-typing` as a linear type constraint after desugaring.

=== Decision

Enforce the Godley invariant as a *pre-desugar structural check* in
`oikos-check::godley_check`.

=== Consequences

* The check runs even while the Ephapax desugar/type-check path is blocked.
  Economist users get meaningful error messages immediately.
* Error messages can be expressed in economic vocabulary ("sector X column
  does not sum to zero") rather than Ephapax type system vocabulary.
* The check is conservative: it checks sign-pattern balance, not
  value-magnitude balance.  Full magnitude checking will be added once the
  Ephapax integration is complete.
* Duplication risk is low because the post-desugar Ephapax check will catch
  anything the pre-check misses.

== DR-004 — Instrument typestate reachability check

*Date:* 2026-04-11 +
*Status:* accepted

=== Context

Financial instrument typestate machines can contain unreachable states or
states from which no terminal state is reachable.
These are bugs in the model definition, not valid instrument designs.

=== Decision

Implement BFS-based reachability checks in `oikos-check::instrument_check`:

1. Every declared state must be reachable from the initial state.
2. Every non-terminal state must have at least one path to a terminal state.

Terminal states are `Settled` and `Void` (or custom states with no
outgoing transitions).

=== Consequences

* Models with dead states in their instrument typestate machines are
  rejected at compile time.
* The BFS is linear in the number of states + transitions; performance
  is not a concern for realistic instrument definitions.

== DR-005 — Forge strategy: GitHub primary, GitLab downstream mirror

*Date:* 2026-04-11 +
*Status:* accepted — 2026-04-11

=== Context

Oikos was initially specified as GitLab-primary.
On review, every project Oikos depends on or integrates with (Ephapax,
Betlang, Eclexia, Julia the Viper) is hosted on GitHub under
`hyperpolymath`.  The estate toolchain (hypatia-scan, gitbot-fleet,
scorecard, the mirror system) is wired to GitHub.
GitLab-primary would require maintaining a parallel CI pipeline with no
audience or tooling benefit.

=== Decision

Oikos follows the estate standard: *GitHub-primary*.

* Canonical remote: `https://github.com/hyperpolymath/oikos`
* GitLab (`https://gitlab.com/hyperpolymath/oikos`) is a downstream mirror,
  populated by the estate hub-and-spoke mirror system.
* CI/CD will use GitHub Actions (`.github/workflows/`) rather than
  `.gitlab-ci.yml`; the existing `.gitlab-ci.yml` scaffold can be removed
  once GitHub Actions workflows are in place.

=== Consequences

* Consistent with the estate-wide rule; no special case to maintain.
* All estate tooling (hypatia-scan, scorecard, gitbot-fleet) applies
  without adaptation.
* Dependency references to Ephapax and other `hyperpolymath` projects
  work without cross-forge coordination.
* GitLab still receives the repo via the downstream mirror, satisfying
  any EU-institution preference for GitLab access.

== DR-006 — Betlang interoperability: stochastic monetary expressions

*Date:* 2026-04-11 +
*Status:* proposed

=== Context

Real SFC models require scenario analysis: "what if wages grow at this
distribution of rates?"  Currently `MoneyExpr` is limited to
`Literal | Balance | Fraction` — all deterministic.
Betlang (`nextgen-languages/betlang`) is a ternary probabilistic DSL with
Bayesian inference, MCMC sampling, and 14 number systems for uncertainty.

A secondary motivation is **auction theory**.
Treasury/gilt auctions are financial instruments in SFC models: the government
issues a bond through an auction mechanism, and the auction price determines
the balance-sheet entry.
This maps naturally to an extended instrument typestate
`Draft → OnAuction → Issued → Settled | Void`, where Betlang models the
distribution of clearing prices and Oikos records the resulting flows.

=== Proposed decision

Add `MoneyExpr::Stochastic(BetExpr)` as a fourth variant.
The type constraint holds in all branches: if all branches of a `BetExpr`
resolve to `Money[C]` for the same currency `C`, the whole expression is
`Money[C]`.  This is enforceable at the Ephapax level because both DSLs
share the same desugar target.

An `oikos-sim` crate wraps betlangiser (the existing Rust bridge to Betlang)
and runs the Oikos interpreter N times under sampled parameter regimes,
returning distributional results over end-of-period stock balances.

=== Separation of concerns

Oikos owns the accounting structure (Godley invariant, linearity, period
regions).  Betlang owns the uncertainty (distributions, inference, sampling).
The Godley column-sum rule continues to hold as a hard constraint in every
sampled branch — stochasticity cannot violate the accounting identity.

=== Consequences

*Positive:*

* Economists can express parameter uncertainty without leaving the Oikos DSL.
* The betlangiser Rust crate already exists; the integration surface is
  well-defined (one new `MoneyExpr` variant + a simulation runner).
* Auction models become first-class: instrument typestate + stochastic amounts
  + Godley accounting in a single model.

*Negative / open questions:*

* `MoneyExpr::Stochastic` must carry type information at the point of use,
  not just at evaluation time.  The Betlang expression must be typed
  `Bet[Money[C], Money[C], Money[C]]` before Oikos accepts it.
  This requires Betlang to have a typed expression representation that Oikos
  can inspect, or a narrowing convention at the interop boundary.
* Betlang is currently Racket/Julia; betlangiser provides a Rust FFI bridge.
  The bridge API will need to expose typed money expressions specifically.
* Simulation outputs (distributions over stock balances) are not yet
  representable in the `oikos-syntax` AST.  A separate `oikos-sim` output type
  is needed.

=== Alternatives considered

* *Julia/ModelingToolkit with stochastic parameters:* rejected.  MTK does not
  enforce linearity or accounting identities statically; violations would be
  runtime errors (see DR-001).
* *Inline distribution literals in Oikos:* considered.  A mini-distribution
  sub-language in Oikos itself would avoid the Betlang dependency.
  Rejected because Betlang's Bayesian inference and 14 number systems are
  substantially more powerful than what Oikos would re-implement.

== DR-007 — Eclexia interoperability: resource-constrained SFC models

*Date:* 2026-04-11 +
*Status:* proposed

=== Context

Eclexia (`nextgen-languages/eclexia`) is a statically-typed compiled language
that treats energy, carbon, time, and memory as first-class phantom type
parameters with compile-time dimensional analysis and an LP shadow-price engine.

Post-Keynesian and ecological economics increasingly demand that SFC models
account for physical resource flows alongside monetary flows.
An Eclexia integration would let an Oikos model answer questions like:
"what is the carbon cost of this monetary transfer?" or
"under a carbon budget of X, which transfer schedule minimises economic
disruption?" (using Eclexia's LP duality).

=== Proposed decision

Annotate `TransferExpr` with optional Eclexia resource annotations:

[source,oikos]
----
transfer wages → deposits {
    amount: 2_500.00 GBP
    @requires energy: 0.05 kWh
    @requires carbon: 0.02 kgCO2e
}
----

The annotations are transparent to the Oikos type checker (they do not affect
monetary types or linearity) but are forwarded to Eclexia's resource checker,
which verifies that the annotated totals satisfy declared resource budgets.

Dimensional analysis is shared: both Oikos (currencies as phantom types) and
Eclexia (physical units as phantom types) use the same conceptual mechanism
via Ephapax phantom-type parameters.

=== Consequences

*Positive:*

* Enables true ecological SFC modelling — monetary and physical resource flows
  in one coherent model.
* Eclexia's LP shadow-price engine can supply the marginal resource cost of
  relaxing a constraint (e.g. "relaxing the carbon budget by 1 tonne permits
  £500 of additional wage payments").
* No changes to the Oikos core type system; annotations are a pass-through.

*Negative / open questions:*

* Eclexia's resource types and Oikos's monetary types do not currently share
  an Ephapax encoding.  An adapter layer is needed.
* The Eclexia LP solver is a runtime component; Oikos's model compilation is
  static.  The integration point is post-compilation simulation, not
  compile-time checking.
* Resource annotation syntax needs to be defined and reserved in the Oikos
  lexer/parser without conflicting with existing keywords.

== DR-008 — Julia the Viper interoperability: data/control separation

*Date:* 2026-04-11 +
*Status:* proposed

=== Context

Julia the Viper (`julia-the-viper`) enforces Harvard Architecture separation
between a Turing-incomplete Data Language (total, no effects, addition and
negation only) and a Turing-complete Control Language (loops, conditionals, I/O).

Oikos has an analogous structure:

* `MoneyExpr` (the amount sub-language) is pure and should be total —
  a `balance(account)` expression always terminates with a monetary value.
* The model body (sequence of `transfer` / `close` / `convert` expressions)
  is the control layer; it is Turing-complete if loops or conditionals are
  ever added.

JtV's purity levels (`@total` / `@pure` / `@impure`) map naturally:
`MoneyExpr` is `@total`, `transfer` bodies are `@pure` (no I/O),
future simulation loops would be `@impure`.

=== Proposed decision

Adopt JtV's purity vocabulary as annotation syntax in Oikos's type error
messages and documentation, without a hard runtime dependency.

If JtV's Lean 4 proof infrastructure is available, the totality of
`MoneyExpr` evaluation (every expression terminates with a `Money[C]` value)
can be discharged as a JtV-style formal proof rather than a custom Oikos proof.

=== Consequences

*Positive:*

* Conceptual alignment: SFC economists and JtV users share the intuition that
  "amounts are data, transfers are control."
* The JtV Lean 4 formalisation of totality avoids duplicating a proof that
  already exists in the JtV proof infrastructure.
* Shared vocabulary makes cross-project documentation clearer.

*Negative / open questions:*

* JtV's Data Language is deliberately limited to addition and negation.
  Oikos `MoneyExpr` includes `balance(account)` and `fraction(ratio, account)`,
  which are read operations, not just arithmetic.  The totality argument
  requires extending JtV's proof to cover these read operations.
* The integration is documentation/proof-level, not code-level.
  No shared crate dependency is anticipated.

== DR-009 — Pre-Ephapax type checker: three-mechanism sufficiency

*Date:* 2026-04-11 +
*Status:* accepted

=== Context

Phase 6 (Ephapax desugar + type checking) is blocked on the `subst` lemma
closing in `ephapax-typing`.  The question arose: is the set of type-theoretic
mechanisms Ephapax provides — phantom types, region-scoped immutable values
(Tofte–Talpin regions), and linear types — actually sufficient for all the
semantic guarantees Oikos requires?  Or does Oikos need anything beyond them?

=== Decision

The three mechanisms are *exactly sufficient*.  No additional mechanism is
needed.  The mapping is:

[cols="2,2,3"]
|===
| Oikos requirement | Ephapax mechanism | Notes

| Stock ≠ Flow
| Phantom type `Account<Kind>`
| Different `Kind` phantom parameters are incompatible types at compile time.

| GBP ≠ USD (currency mixing)
| Phantom type `Money<Currency>`
| Currency codes become phantom type parameters; mixing is a type error.

| Period scoping (FY2024 values cannot bleed into FY2025)
| Tofte–Talpin regions
| Each fiscal period is a region; balance values are region-scoped.
  `close from P into Q` is a region transfer, not a copy.

| Godley column-sum invariant
| _Already proved pre-desugar_
| Handled entirely in Phase 4 (`godley_check`); Ephapax does not need to
  re-prove it.

| Instrument lifecycle (`Draft → OnAuction → Issued → Settled\|Void`)
| Linear typestate
| Typestate is specialised linear typing: the old state type is consumed,
  a new state type is returned.  Ephapax's linear types subsume this directly.

| Linear transfers (source consumed, cannot be double-spent)
| Linear types
| Direct application: transfer consumes the value at the source linearly.
|===

The `subst` lemma that is currently blocking `ephapax-typing` is specifically
the lemma required to typecheck `close from P into Q` (a value moving from
one region to another).  Closing that lemma unblocks the most important
remaining piece of Oikos.

=== Consequences

* No exotic type theory is needed.  The existing Ephapax design is the right
  level of expressiveness for Oikos — no dependent types, no refinement types,
  no substructural systems beyond linear.
* The three pre-desugar phases that already exist (cross-reference, currency,
  instrument) are correctly scoped: they catch what can be caught without
  type-theoretic machinery, leaving the genuine type-checking to Ephapax.
* Decision to return to Ephapax work is confirmed: completing `subst` is the
  single most valuable thing to do next for both Ephapax and Oikos.

== DR-010 — MPL-2.0 licence (supersedes DR-002)

*Date:* 2026-05-22 +
*Status:* accepted

=== Context

DR-002 (2026-04-11) selected EUPL-1.2 as an explicit exception to the
estate-wide MPL-2.0 default.  Two follow-on findings made the exception
costly:

* *PMPL ↔ EUPL compatibility never resolved.*  The flag raised in
  DR-002's Consequences (legal review of PMPL ↔ EUPL compatibility
  before v1.0) remained open.  Oikos depends on Ephapax, which is
  PMPL-1.0-or-later with MPL-2.0 fallback — a binary that statically
  links Ephapax under EUPL-1.2 has an unresolved combinatorial-licence
  question that we are not equipped to answer without paid legal advice.
* *Distribution friction across the estate.*  Every adjacent
  hyperpolymath project the user might want to combine with Oikos
  (cartridges, analyzers, bot-integration) is MPL-2.0; an EUPL-licensed
  core forces every cross-project consumer to re-do the compatibility
  analysis.

=== Decision

Adopt MPL-2.0 across Oikos, superseding DR-002.  The LICENSE file at
the repository root was migrated on 2026-05-22 (commit `5ea61de`); this
DR records the rationale and supersedes DR-002's explicit-exception
status.

=== Consequences

* The PMPL ↔ EUPL compatibility flag is closed — Ephapax (PMPL/MPL-2.0)
  and Oikos (MPL-2.0) compose without question.
* EU translations and the EUPL governance hook are lost.  The estate
  accepts this trade-off: economist users were never going to read the
  Slovenian translation of the licence in practice, and MPL-2.0 is
  itself OSI-approved with broad recognition.
* SPDX headers across the codebase need to flip from `EUPL-1.2` to
  `MPL-2.0`; this is a mechanical sweep.
* The historical `LICENSES/EUPL-1.2.txt` file is kept for a transition
  period as a record of the prior choice and removed in a separate
  cleanup.

|===
| ID | Question | Priority

| Q-1
| *Ephapax error translation:* how should Ephapax type errors be

  Options: post-processing the error message string; a dedicated
  `oikos-diagnostics` crate; teaching Ephapax to carry domain metadata.

| Q-2
| *Decimal representation:* `MoneyLiteral.amount` is stored as a `SmolStr`.
  Before arithmetic (in the type checker or desugaring), we need a
  fixed-point or arbitrary-precision decimal type.
  Options: `rust_decimal`, `bigdecimal`, custom.
| Medium

| Q-3
| *PMPL ↔ EUPL licence compatibility:* legal review required before
  distributing Oikos (EUPL-1.2) as a binary that statically links
  Ephapax (MPL-2.0).
| High

| Q-4
| *IDE integration:* an LSP server would make Oikos useful to economists
  working in standard editors.  The Ephapax project has `ephapax-lsp`;
  Oikos can either extend it or ship `oikos-lsp`.
| Low (post-v1.0)
|===
