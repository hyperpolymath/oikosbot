== "A code-analysis App that adds an ecological & economic review lens to PRs"

[quote, README.adoc]
____
A code-analysis GitHub/GitLab App that adds ecological and economic review to
pull requests.
____

How this is implemented::
The analysis core is the Rust workspace under link:crates/[`crates/`]:
`oikosbot-cli` (the `oikosbot` binary), `oikosbot-analysis` (the analysis
engine + eco/econ heuristics), `oikosbot-metrics` (shared metric/result types),
`oikosbot-sarif` (SARIF report generation), and `oikosbot-eclexia` (Eclexia
`.ecl` policy evaluation). This workspace builds and tests cleanly and is the
most complete surface; it was extracted from the former
`gitbot-fleet/bots/sustainabot/` and renamed `sustainabot-*` â†’ `oikosbot-*`.

Caveat::
The default `cargo` workspace excludes `crates/oikosbot-fleet` (the optional
gitbot-fleet bridge) so OikosBot builds standalone with no dependency on
gitbot-fleet.

== "AffineScript webhook receiver with HMAC verification"

[quote, README.adoc]
____
The AffineScript webhook receiver is in flight.
____

How this is implemented::
The receiver lives in link:bot-integration-affine/src/[`bot-integration-affine/src/`].
HMAC-SHA256 webhook signature verification is real:
`bot-integration-affine/src/Webhook.affine` strips the GitHub `sha256=` prefix
and calls `hmac_sha256_verify`, which is implemented constant-time by the
`hpm-crypto` RSR (`hpm_crypto_hmac_sha256_verify`) reached through AffineScript's
`extern fn` mechanism (`src/Externs.affine`). The orchestration entry points are
present in `src/Main.affine` (config load, HTTP listener extern, router dispatch),
`src/Router.affine` (path dispatch for `/health`, `/webhooks/github`,
`/webhooks/gitlab`, `/metrics`), `src/Oikos.affine` (webhook â†’ analyse â†’ comment),
`src/Analysis.affine`, and `src/Report.affine` (Markdown PR-comment formatting).
A smoke test lives at `bot-integration-affine/test/Smoke.affine`
(driver: `bot-integration-affine/check.sh`).

Caveat::
The two upstream AffineScript standard-library prerequisites have landed (`json`
RSR rewire, affinescript#421; `Http` server RSR rewire, affinescript#425). The
webhook â†’ analyse â†’ comment orchestration is now wired in source:
`src/Payload.affine` extracts the PR fields (stdlib `json` lazy descent) and
`handle_github_webhook` runs `Oikos.handle_pr_event` (auth â†’ analyse â†’ comment),
returning `200`/`500`/`202` by outcome. What remains is *binding the HTTP
listener* (`Sub.http_server`), which is gated on the stdlib `Http::Server` host
type (see `tea/Runtime.affine`), so nothing runs end-to-end yet. These
AffineScript changes are *unverified* â€” no AS toolchain was available to compile
them. HMAC verification can be bypassed only via the dev-only
`OIKOS_ALLOW_UNVERIFIED_WEBHOOKS` escape hatch, which must never be set in
production.

== "Haskell eco/econ/quality analysis engine"

[quote, README.adoc]
____
The Haskell analyzer scaffold builds independently.
____

How this is implemented::
link:analyzers/code-haskell/[`analyzers/code-haskell/`] is a Cabal project
(`oikos-analyzer.cabal`) with the ecological modules `Eco.Carbon`, `Eco.Energy`,
`Eco.Pareto`, `Eco.Resource`, `Eco.Analysis`; the quality modules
`Quality.Complexity`, `Quality.Coupling`, `Quality.Coverage`, `Quality.Debt`;
and shared types in `Types.Metrics` / `Types.Report`. Entry point is `app/Main.hs`;
tests are in `test/Main.hs`.

Caveat::
The analyzer is *heuristic*, not ontology-driven â€” it estimates carbon/energy/
complexity from code-shape signals, it does not perform formal semantic analysis
against an ontology. The RDF/OWL ontology at `databases/ontology/eco.ttl` is seed
data for VeriSimDB's semantic witness (see below), not an input the Haskell
analyzer consumes today. The analyzer service is not yet wired to the AffineScript
receiver.

== "Policy engine: Datalog + DeepProbLog, plus Eclexia policies"

[quote, README.adoc]
____
policy-engine/ â€” Datalog and DeepProbLog policy rules; policies/ â€” Eclexia
(.ecl) policy definitions.
____

How this is implemented::
`policy-engine/datalog/eco_rules.dl` holds the deterministic rules and
`policy-engine/deepproblog/eco_problog.pl` the probabilistic (learned) rules.
The Eclexia resource policies in link:policies/[`policies/`]
(`carbon_budget.ecl`, `energy_threshold.ecl`, `memory_efficiency.ecl`,
`security_sustainability.ecl`) are evaluated by the `oikosbot-eclexia` crate.

Caveat::
The DeepProbLog rules and the praxis (learning) loop are design-stage: the
probabilistic predicates and neural-predicate declarations exist as logic, but
there is no trained model or live feedback ingestion wired in. The knowledge-graph
bindings in `eco_problog.pl` were retargeted from SPARQL-over-Virtuoso to
VCL-over-VeriSimDB at the comment/binding level only (no rule logic changed).

== "Single data layer: VeriSimDB"

[quote, README.adoc]
____
OikosBot persists analysis results â€¦ in a single identity-consonance store,
VeriSimDB.
____

How this is implemented::
link:config/oikos.yaml[`config/oikos.yaml`] declares a single `verisimdb:`
datastore block (REST endpoint + octad witness map), replacing the previous
`arangodb:` + `virtuoso:` pair. The eco/econ ontology seed for the semantic
witness lives at link:databases/ontology/eco.ttl[`databases/ontology/eco.ttl`].
The integration descriptor is
link:.machine_readable/integrations/verisimdb.a2ml[`.machine_readable/integrations/verisimdb.a2ml`].
VeriSimDB maps cleanly onto OikosBot's needs: the **graph + document** witnesses
subsume ArangoDB; the **semantic** witness subsumes Virtuoso/RDF.

Caveat::
This is *design + configuration only*. VeriSimDB is `v0.1.0` and OikosBot's
runtime client is **deferred/gated** â€” no live persistence happens yet. The
client (a Zig-FFI connector mirroring the `hpm-*` pattern, or a REST client from
the AffineScript transport) is tracked on the link:ROADMAP.adoc[roadmap] and is
gated on VeriSimDB maturing and on the AffineScript transport (affinescript
#421/#425 integration). The legacy ArangoDB + Virtuoso assets were removed
(recoverable in git history).

== Known Gaps

[CAUTION]
====
*OikosBot is not deployed in production.* The pieces below are honest gaps, not
finished features.
====

[CAUTION]
====
*The HTTP listener is not bound.* The webhook â†’ analyse â†’ comment orchestration
is wired in source (`src/Payload.affine` + `handle_github_webhook` â†’
`Oikos.handle_pr_event`), but binding the listener that feeds request bodies to
the handlers is gated on the stdlib `Http::Server` host type, so nothing runs
end-to-end yet. The AffineScript changes are unverified (no AS toolchain here).
====

[CAUTION]
====
*The analyzer is heuristic, not semantic.* Carbon/energy/complexity scores are
code-shape estimates; there is no formal ontology-driven analysis, and the
Haskell analyzer is not yet connected to the receiver.
====

[CAUTION]
====
*The VeriSimDB data layer is design-only.* No runtime persistence happens yet;
the client is deferred and gated on VeriSimDB maturity and the AffineScript
transport. See the link:ROADMAP.adoc[roadmap].
====
